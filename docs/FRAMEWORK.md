# xCSG Value Measurement Framework

**CONFIDENTIAL** — Alira Health xCSG Initiative

---

## First Principle

**Value = Outcome / Time**

In consulting, value improves when the outcome is better OR the time is less. The xCSG model claims to improve both simultaneously through a three-legged flywheel. This framework measures that claim by comparing xCSG delivery against Alira Health legacy operations on both dimensions, per deliverable, over time.

---

## The Three Flywheel Legs (What We Measure)

| Leg | What it claims | How it improves Value |
|-----|---------------|----------------------|
| **Machine-First Operations** | AI handles assembly; humans handle judgment | Compresses **Time** (denominator) and expands **Scope** (numerator) |
| **Senior-Led Specialized Engagement Ops** | Domain experts author deliverables directly, not through intermediary layers | Improves **Outcome** (numerator) through higher first-pass quality and zero signal dilution |
| **Proprietary Knowledge** | Alira registries, institutional data, and codified methodologies create un-replicable insights | Improves **Outcome** (differentiates) AND compresses **Time** (higher starting point) |

The flywheel thesis: each leg feeds the others. AI makes the specialist more productive, the specialist's judgment makes the AI output better, and both generate proprietary knowledge that makes the next engagement start from a higher baseline. The measurement framework tests whether this compounding is actually happening.

---

## Survey Design

**Administration**: Human-collected (external tracker, not AI-triggered). Weekly or per-deliverable — frequency TBD.

**Principles**: Zero free text. All categorical or quantitative. Every question either captures an observable fact (xCSG actual) or a calibrated professional norm (legacy baseline). Every question maps to the value equation or a flywheel leg.

**Total**: 20 questions across 6 sections.

---

### SECTION A — CONTEXT

These set the baseline for like-with-like comparison.

| # | Question | Options | Maps to |
|---|----------|---------|---------|
| A1 | Deliverable type | CDD / Competitive landscape / Financial model / Market access / Proposal / Call prep brief / Presentation / KOL mapping | Grouping variable |
| A2 | Engagement stage | New business (pre-mandate) / Active engagement / Post-engagement (follow-on) | Commercial context |

**Why A2 is a killer**: A deliverable that wins new business has fundamentally different value than one fulfilling an existing SOW. Proposals that convert are the ultimate outcome signal.

---

### SECTION B — MACHINE-FIRST OPERATIONS

Tests whether AI compresses the assembly layer.

| # | Question | xCSG (actual) | Legacy (norm) | Maps to |
|---|----------|---------------|---------------|---------|
| B1 | Starting point: Did you build the final deliverable from an AI-generated draft or from a blank page? | From AI draft / Mixed (AI structure, manual content) / From blank page | Always blank page (by definition) | Machine-First adoption |
| B2 | Research throughput: How many distinct data sources were synthesized in this deliverable? | 1-3 / 4-7 / 8-12 / 13+ | 1-3 / 4-7 / 8-12 / 13+ | Scope × Machine-First |
| B3 | Assembly ratio: What percentage of data collection and initial structuring was AI-performed? | >75% AI / 50-75% / 25-50% / <25% | 0% (by definition) | Time × Machine-First |
| B4 | Hypothesis approach: Was this deliverable structured around a pre-formed hypothesis or built through open-ended discovery? | Hypothesis-first (tested a specific thesis) / Hybrid (hypothesis emerged during work) / Discovery-first (open-ended research) | Always discovery-first (by definition) | Machine-First × Outcome framing |

**Why B2 is a killer**: If xCSG consistently synthesizes 8-12 sources where legacy does 3-4, the AI isn't just faster — it produces a fundamentally more informed deliverable. That's Outcome improvement, not just Time compression.

**Why B3 is a killer**: Paired with Time data (Section E), it answers: "We moved X% of assembly from humans to machines — and delivery took Y fewer days." The relationship between B3 and E1-E2 IS the machine-first value proof.

**Why B4 is a killer**: The hypothesis-first pattern is the behavioral signature of the xCSG model. All three proof cases (Apposite, J&J, Solventum) started with a pre-formed thesis amplified by machine-first synthesis. If xCSG deliverables consistently score "Hypothesis-first" where legacy scores "Discovery-first," it proves the operating model produces a fundamentally different kind of output — one that leads with insight rather than assembling toward it.

---

### SECTION C — SENIOR-LED SPECIALIZED ENGAGEMENT OPERATIONS

Tests whether domain experts author deliverables directly, with judgment concentrated on high-value work.

| # | Question | xCSG (actual) | Legacy (norm) | Maps to |
|---|----------|---------------|---------------|---------|
| C1 | **Specialization match**: Is the person who did the primary analytical work a recognized domain specialist in the subject matter of this deliverable? | Deep specialist in this TA/methodology / Adjacent expertise / Generalist | Deep specialist / Adjacent expertise / Generalist | Senior-Led × Outcome quality |
| C2 | **Directness**: Did the domain expert AUTHOR the deliverable or only REVIEW it? | Expert authored (with AI assist) / Expert co-authored (shared with team) / Expert reviewed only | Expert authored / Expert co-authored / Expert reviewed only | Senior-Led × Signal dilution |
| C3 | **Judgment concentration**: What percentage of the expert's time on this deliverable was spent on high-value work (analysis, interpretation, strategic framing) vs. low-value assembly (data gathering, formatting, structuring)? | >75% judgment / 50-75% / 25-50% / <25% | >75% judgment / 50-75% / 25-50% / <25% | Senior-Led × Expert leverage |
| C4 | Client revision rounds before acceptance | 0 / 1 / 2 / 3+ | 0 / 1 / 2 / 3+ | Outcome (quality proof) |
| C5 | Did the client expand scope or request additional work after seeing this deliverable? | Yes, expanded scope / Yes, new engagement / No / Not yet delivered | Yes, expanded / Yes, new engagement / No / N/A | Outcome (commercial proof) |

**Why C1 is a killer**: The thesis is that xCSG puts the RIGHT expert on the work, not just A senior person. Legacy often assigns whoever is available because the right specialist is oversubscribed. xCSG's claim: one specialist + AI covers more engagements, so the specialization match improves.

**Why C2 is a killer**: This IS the senior-led thesis in one question. Authored = judgment embedded directly, zero interpretation layers. Reviewed = judgment filtered through someone else's understanding. The gap between "authored" and "reviewed" is the gap between the two models.

**Why C3 is a killer**: A specialist billing at $500/hr spending 80% of time on judgment vs. 30% — same person, same rate, radically different value per hour to the client. The ratio shift IS the productivity story.

**The narrative C1 through C5 tell together**: The RIGHT person (C1) did the work DIRECTLY (C2), concentrating their time on what ONLY THEY can do (C3), producing a result that was RIGHT THE FIRST TIME (C4) and SO RIGHT the client wanted MORE (C5).

---

### SECTION D — PROPRIETARY KNOWLEDGE

Tests whether Alira's institutional data creates un-replicable value — the moat.

| # | Question | xCSG (actual) | Legacy (actual/norm) | Maps to |
|---|----------|---------------|---------------------|---------|
| D1 | Does this deliverable contain specific data points from Alira proprietary sources (registries, prior engagement findings, institutional methodologies) NOT available from public sources? | Yes / No | Yes / No | Proprietary Knowledge (binary test) |
| D2 | Did this deliverable build on reusable knowledge assets (frameworks, prior analyses, codified methodologies) from previous Alira engagements? | Yes, directly reused and extended / Yes, provided useful starting context / No, built from scratch | Yes, directly shaped / Yes, useful context / No | Knowledge compounding |
| D3 | Could a competitor with the same AI tools but WITHOUT Alira's proprietary data and institutional expertise have produced an equivalent deliverable? | No — proprietary inputs were decisive / Partially — they'd miss key insights / Yes — all inputs were publicly available | (same question, same scale) | THE MOAT TEST |

**Why D3 is a killer**: If the answer is consistently "yes, a competitor could match it," the proprietary leg isn't contributing and you're in the commodity trap. One question that tests the entire strategic thesis. Worth the slight subjectivity because the senior who produced the work is uniquely qualified to assess what went into it.

---

### SECTION E — TIME (The Denominator)

xCSG observed, legacy estimated as calibrated professional norms.

| # | Question | xCSG (actual) | Legacy (norm) | Maps to |
|---|----------|---------------|---------------|---------|
| E1 | Calendar days from start to accepted delivery | 1 / 2-3 / 4-5 / 6-10 / 11-20 / 20+ | 1 / 2-3 / 4-5 / 6-10 / 11-20 / 20+ | Time (days) |
| E2 | Team size — people who touched this deliverable | 1 / 2 / 3 / 4+ | 1 / 2 / 3 / 4+ | Time (people) |

**Person-days** = midpoint(E1) × midpoint(E2). Computed, not asked. This is the denominator in the value equation.

**Why calibrated norms, not counterfactual estimates**: The legacy column asks "what is Alira's TYPICAL timeline for this deliverable type" — not "guess how long this specific task would have taken." Seniors know these norms from years of experience. Bounded categorical answers prevent wild estimates.

---

### SECTION F — VALUE CREATION

The value equation in one question.

| # | Question | Options | Maps to |
|---|----------|---------|---------|
| F1 | Would this deliverable have been feasible in the legacy model — same scope, same quality, same timeline? | Not feasible — scope or timeline was only possible with AI / Feasible but at 2x+ the cost and time / Feasible at similar cost — xCSG provided marginal benefit / Legacy would have been more effective | Holistic value comparison |
| F2 | Could a component of this deliverable (a framework, dataset, analysis template) be reused as a standardized offering for similar clients? | Yes, largely as-is / Yes, with moderate customization / No, fully bespoke | Productization signal |

**Why F1 is a killer**: The top answer ("not feasible") identifies deliverables that xCSG CREATES — work that wouldn't exist in the legacy model. That's not efficiency improvement; it's value creation. The distribution of answers to F1 is the headline for the board.

**Why F2 matters**: This question generates early signal for the Agent-as-a-Service (AaaS) strategic option. If a significant percentage of deliverables contain components that could be offered as standardized gate-check assessments, the AaaS path has evidence behind it. This is not a commitment to productize — it's data that informs the strategic discussion.

## Two-Tier Administration Design

The survey is administered by a PMO helper who orchestrates data collection without needing domain expertise.

### Tier 1 — PMO Helper Collects (observable facts, no judgment required)

| Question | Source | Time |
|----------|--------|------|
| A1: Deliverable type | Project tracking | <1 min |
| A2: Engagement stage | SOW/project records | <1 min |
| E1: Calendar days | Date tracking (start → accepted delivery) | <1 min |
| E2: Team size | Staffing records | <1 min |
| C4: Revision rounds | Version history / email trail | <1 min |
| C5: Scope expansion | SOW changes / follow-on requests | <1 min |

**Legacy baseline**: Auto-populated from a reference norms table (see below), editable if a specific case warrants override.

**Total Tier 1 time**: ~5 minutes per deliverable.

### Tier 2 — Expert Self-Reports (judgment required, streamlined)

The remaining 14 questions (B1-B4, C1-C3, D1-D3, F1-F2) require the domain expert who produced the deliverable. These are delivered as a short digital form (unique link per deliverable), pre-populated with context (deliverable type, client, dates, team).

**Target**: <3 minutes for the expert. All categorical, zero free text, single page.

**Workflow**: PMO helper creates the deliverable record (Tier 1) → generates a unique expert link → sends to the expert → expert clicks through 14 dropdowns → record marked complete → metrics auto-compute.

### Legacy Norms Reference Table

Rather than asking each expert to estimate legacy timelines per deliverable, a one-time reference table is maintained and validated by senior leaders:

| Deliverable Type | Typical Calendar Days | Typical Team Size | Typical Revision Rounds |
|-----------------|----------------------|-------------------|------------------------|
| CDD | 11-20 days | 3 people | 2 rounds |
| Competitive landscape | 11-20 days | 3 people | 2 rounds |
| Financial model | 6-10 days | 2 people | 1 round |
| Market access | 11-20 days | 3 people | 2 rounds |
| Proposal | 4-5 days | 2 people | 1 round |
| Call prep brief | 2-3 days | 1 person | 1 round |
| Presentation | 4-5 days | 2 people | 2 rounds |
| KOL mapping | 11-20 days | 3 people | 2 rounds |

These auto-populate when the PMO helper selects a deliverable type. Override is available per record. The table is editable by administrators and should be recalibrated quarterly.

## Data Usage — Computing the Value Multiplier

### Per-Deliverable Computation

```
Person-Days_xCSG    = midpoint(E1) × midpoint(E2)
Person-Days_Legacy  = midpoint(E3) × midpoint(E4)
Effort_Ratio        = Person-Days_Legacy / Person-Days_xCSG

Quality_Ratio       = Revisions_Legacy(C4) / Revisions_xCSG(C4)
                      (if xCSG revisions = 0, use 0.5 as denominator)

Value_Multiplier    = Effort_Ratio × Quality_Ratio
```

### Midpoint Mapping for Categorical Ranges

| Range | Midpoint |
|-------|----------|
| 1 day | 1 |
| 2-3 days | 2.5 |
| 4-5 days | 4.5 |
| 6-10 days | 8 |
| 11-20 days | 15 |
| 20+ days | 25 |

| Team size | Midpoint |
|-----------|----------|
| 1 | 1 |
| 2 | 2 |
| 3 | 3 |
| 4+ | 5 |

### Worked Example

A competitive landscape deliverable:
- xCSG: 4-5 days, 1 person = 4.5 person-days, 0 revision rounds
- Legacy norm: 11-20 days, 3 people = 45 person-days, 2 revision rounds
- Effort Ratio = 45 / 4.5 = **10x**
- Quality Ratio = 2 / 0.5 = **4x**
- Value Multiplier = 10 × 4 = **40x**

This means: xCSG produced a deliverable that was 10x less effort AND 4x better quality. Combined value: 40x.

### Flywheel Leg Scores (per checkpoint)

Each leg gets a composite score derived from its survey questions:

**Machine-First Score**: Average of B1 (AI draft adoption: 1.0/0.5/0), B2 (sources ratio: xCSG/Legacy), B3 (assembly ratio mapped to 0-1 scale), B4 (hypothesis-first: 1.0/0.5/0)

**Senior-Led Score**: Average of C1 (specialization match: 1.0/0.5/0), C2 (authored=1.0/co-authored=0.5/reviewed=0), C3 (judgment concentration ratio: xCSG/Legacy)

**Proprietary Knowledge Score**: Average of D1 (proprietary data: 1/0), D2 (institutional reuse: 1.0/0.5/0), D3 (moat: 1.0/0.5/0)

All three scores range 0 to 1. Trending toward 1.0 = that leg is fully engaged. Flat at 0 = that leg isn't contributing.

## Incremental Dashboard Design

The dashboard builds progressively as data accumulates. Each checkpoint adds new panels and deepens the analysis.

### Checkpoint 1 (Weeks 1-2): Scorecard

**Data available**: 1-3 deliverables. Not enough for trends.

**Dashboard — 1 panel**:

| Panel | Content | What it proves |
|-------|---------|---------------|
| **Scorecard** | Table: each deliverable as a row, showing: Type, xCSG person-days, Legacy person-days, Effort Ratio, Revisions (xCSG vs Legacy), Value Multiplier | "It's working. Here are the first data points." |

**Supplementary**: F1 distribution (even with 1-3 data points, "not feasible in legacy" is a powerful statement)

**Audience**: Pioneers (motivation), PJ (early signal)

---

### Checkpoint 2 (Week 4): First Comparisons

**Data available**: 5-8 deliverables. Enough for side-by-side comparisons.

**Dashboard — 3 panels**:

| Panel | Content | What it proves |
|-------|---------|---------------|
| **Side-by-Side Bars** | Per deliverable: Legacy person-days (gray bar) vs xCSG person-days (navy bar). Visceral visual compression. | "Look at the gap. Every single deliverable." |
| **Quality Comparison** | Per deliverable: Legacy revision rounds vs xCSG revision rounds. Paired bars. | "Not just faster — better first-pass quality." |
| **Leg Indicators** | Three gauges: Machine-First score, Senior-Led score, Proprietary Knowledge score (averages across all deliverables so far) | "All three legs are engaged (or: here's which leg needs attention)." |

**Audience**: Practice leaders (operational proof), Pioneers (progress)

---

### Checkpoint 3 (Week 8): Trend Lines

**Data available**: 12-20 deliverables. Enough for meaningful trends.

**Dashboard — 5 panels**:

| Panel | Content | What it proves |
|-------|---------|---------------|
| **Value Multiplier Trend** (HERO) | Line chart: Value Multiplier per deliverable over time, with rolling average. X-axis = weeks, Y-axis = multiplier. | "The flywheel is accelerating." If the line slopes up, compounding is real. |
| **Side-by-Side Bars** | Continues from Checkpoint 2, now with more data points | Evidence depth |
| **Quality Trend** | Line chart: xCSG revision rounds over time. Should be flat or declining. | "Quality is consistent or improving." |
| **Flywheel Leg Trends** | Three line charts (or stacked area): Machine-First, Senior-Led, Proprietary Knowledge scores over time | "Each leg is strengthening." Shows which leg drives the most improvement. |

**New metric introduced**: **Compound Signal** — D2 trend over time (% of deliverables where prior Alira engagement work "directly shaped the output"). If this rises, the flywheel is compounding.

**Audience**: All three — Gabriele (trend + financial), Practice leaders (legs), Pioneers (personal growth)

**Data available**: 20-40 deliverables. Statistically meaningful.

**Dashboard — 7 panels**:

| Panel | Content | What it proves |
|-------|---------|---------------|
| **Value Multiplier Trend** (HERO) | Same as Checkpoint 3, now with full pilot data. Add: average multiplier as a bold number. | The headline: "xCSG delivers Nx value." |
| **Side-by-Side Bars** | Full set, grouped by deliverable type | Proves it works ACROSS engagement types (Scaling Gate 1) |

| **Flywheel Leg Dashboard** | Three panels with trends + current scores | Shows maturity of each leg |

| **Seniority Shift** | C1 + C2 comparison: who does the work and how (authored vs reviewed), xCSG vs legacy | Structural proof that the operating model changed |

**New metrics introduced**:
- **Implied Margin Improvement**: (Person-Days_Legacy - Person-Days_xCSG) × Blended Daily Rate = cost avoided per deliverable. Cumulative across pilot.
- **Revenue Attribution**: Deliverables at engagement stage "New business" where C5 = "Yes, new engagement" = pipeline influenced by xCSG.
- **Scaling Readiness**: Map survey data to Scaling Gates 1-5 (multiple types ✓, time reduction ✓, client-invisible quality ✓, transferability TBD, flywheel validation TBD)

**Audience**: Gabriele and the Board. This IS the scaling decision evidence.

## Tracking Spreadsheet Structure

All survey data accumulates in `xCSG_Value_Tracker.xlsx`:

**Sheet 1: Raw Data**
One row per survey response (per deliverable). Columns: Date, Pioneer, A1-A2, B1-B4, C1-C5 (xCSG), C1-C5 (Legacy), D1-D3, E1-E2 (xCSG), E1-E2 (Legacy), F1-F2.

**Sheet 2: Computed Metrics**
Auto-calculated from Raw Data: Person-Days_xCSG, Person-Days_Legacy, Effort_Ratio, Quality_Ratio, Value_Multiplier, Machine-First Score, Senior-Led Score, Proprietary Knowledge Score, Compound Signal.

**Sheet 3: Dashboard**
Charts that update as rows are added to Sheet 1. Panels correspond to the current checkpoint (add panels as checkpoints are reached).

**Sheet 4: Checkpoint Summaries**
Snapshot tables at each checkpoint with key metrics, narrative highlights, and scaling gate status.

## What This Framework Does NOT Measure

Transparency on limitations:

- **Client satisfaction directly**: Revision rounds and scope expansion are proxies, not NPS scores. True client sentiment requires separate feedback.
- **Long-term revenue impact**: Follow-on engagement tracking is lagging data. The framework captures early signals (C5) but not multi-quarter revenue attribution.
- **Individual pioneer performance**: The framework measures the MODEL (xCSG vs legacy), not individual consultants. Avoid using it as a performance evaluation tool.
- **Causation**: The legacy column is always an estimated norm, never an observed parallel. The framework builds a body of evidence, not a controlled experiment.

## Summary

| Element | Design choice | Rationale |
|---------|--------------|-----------|
| 20 questions | All categorical, zero free text | Fully chartable, minimal overhead |
| Paired columns | xCSG (observed) vs Legacy (calibrated norm) | Enables direct comparison |
| Value = Outcome / Time | Effort Ratio × Quality Ratio | First principle of consulting value |
| Three leg scores | Machine-First, Senior-Led, Proprietary Knowledge | Maps every question to the flywheel |
| Incremental dashboard | 4 checkpoints, panels added progressively | Matches data availability to analytical depth |
| External administration | Human-collected, not AI-triggered | Ensures compliance, separates measurement from delivery |
| Two-tier administration | PMO helper (Tier 1: facts) + Expert (Tier 2: judgment) | Minimizes expert burden, ensures compliance |
| Legacy norms reference table | Pre-populated per deliverable type, editable | Removes estimation error, saves time |
